## Introduction

GBI Logistics is a Flutter-based mobile application designed for efficient logistics management, focusing on guide dispatch and reception through a transport cube system. The application follows a clean architecture approach with clear separation of concerns between business logic, UI, and data layers.

## Key Architectural Decisions

### 1. Feature-Based Architecture

The codebase is organized by features rather than layers, promoting modularity and scalability:

```
lib/
├── core/              # Shared utilities and base services
└── features/          # Feature modules
    └── logistics/     # Main logistics feature
```

**Rationale**: This structure allows teams to work on features independently and makes it easier to extract features into separate packages if needed.

### 2. Provider Pattern for State Management

The application uses Provider for dependency injection and state management:

```
MultiProvider(
  providers: [
    ChangeNotifierProvider<AuthProvider>(
      create: (_) => AuthProvider(authService),
    ),
    ChangeNotifierProvider<GuideProvider>(
      create: (_) => GuideProvider(guideService),
    ),
    ChangeNotifierProvider<TransportCubeProvider>(
      create: (_) => TransportCubeProvider(transportCubeService),
    ),
  ],
  child: MaterialApp(...),
)
```

**Rationale**: Provider offers a good balance between simplicity and power, with excellent Flutter integration and minimal boilerplate.

### 3. Service Layer Abstraction

Business logic is encapsulated in service classes that communicate with the API:

- `AuthService` - Authentication and session management
- `GuideService` - Guide operations
- `TransportCubeService` - Transport cube operations
- `HttpService` - Base HTTP communication

**Rationale**: Separating services from providers allows for easier testing and potential reuse across different state management solutions.

### 4. Generic API Response Pattern

A generic `ApiResponse<T>` wrapper handles all API responses consistently:

```
class ApiResponse<T> {
  final bool isSuccessful;
  final String? message;
  final String? messageDetail;
  final T? content;
}
```

**Rationale**: Standardizes error handling and success responses across the entire application.

### 5. Observer Pattern for Lifecycle Management

The `AuthLifecycleObserver` monitors app lifecycle events and refreshes authentication:

```
class AuthLifecycleObserver with WidgetsBindingObserver {
  @override
  void didChangeAppLifecycleState(AppLifecycleState state) async {
    if (state == AppLifecycleState.resumed) {
      // Refresh auth token when app resumes
      await auth.ensureFreshToken();
    }
  }
}
```

**Rationale**: Ensures the user remains authenticated even after backgrounding the app.

### 6. Token Refresh Strategy

Implements proactive token refresh with multiple strategies:
- Periodic checks every 15 minutes
- Refresh 30 minutes before expiration
- Silent login with stored credentials
- Automatic retry on API calls

**Rationale**: Provides a seamless user experience without interrupting workflows.

## Development Environment Setup

### Prerequisites
1. **Flutter SDK** - Version 3.0.0 or higher
2. **IDE** - VS Code or Android Studio recommended
3. **Git** - Version control

### Installation Steps

1. Clone the repository:
```bash
git clone <repository-url>
cd gbi-logistics
```

2. Install dependencies:
```bash
flutter pub get
```

3. Verify Flutter setup:
```bash
flutter doctor
```

4. Run the application:
```bash
flutter run
```

### Environment Configuration

The app supports development and production environments:

```
class ApiConfig {
  static const bool isDevelopment = true;
  static const String devBaseUrl = 'http://100.104.120.121:81';
  static const String prodBaseUrl = 'https://internalapi.gbilogistics.net';
  
  static String get baseUrl => isDevelopment ? devBaseUrl : prodBaseUrl;
}
```

Change `isDevelopment` to `false` for production builds.

## Error Handling Strategy

### Global Error Handling

The app captures unhandled errors at the root level:

```
FlutterError.onError = (FlutterErrorDetails details) {
  FlutterError.presentError(details);
  debugPrint('FlutterError: ${details.toString()}');
};
```

### API Error Codes

Standardized error codes from the API:
- `0`: Success
- `1`: General error
- `2`: Authentication error
- `60`: Session expired

### User-Facing Error Messages

Errors are displayed through:
- `SnackBar` for temporary notifications
- Dialog boxes for critical errors
- Inline validation messages for form inputs

## Performance Considerations

### State Management Optimization

- Loading guards prevent duplicate API calls
- Selective widget rebuilding with `Consumer`
- Debouncing for search inputs

### Network Optimization

- 30-second timeout on all requests
- Request cancellation on navigation
- Pagination for large data sets

### Memory Management

- Proper disposal of providers and controllers
- Image caching for network images
- List virtualization for long lists

## Security Practices

### Authentication
- JWT tokens for API authentication
- Secure storage for credentials
- Automatic token refresh
- Session expiration handling

### Data Protection
- HTTPS for all API communication
- Encrypted storage for sensitive data
- No hardcoded credentials

### Version Control
- Minimum version enforcement
- Optional update notifications
- Force update capability
